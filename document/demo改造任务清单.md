# WeLink Demo 改造任务清单（依据《项目备忘录》）

## 1. 改造目标
- 以现有 Demo 为基础，优先实现备忘录中的 P0 核心要求：可靠性、幂等性、在线/离线路由、基础可观测。
- 保持当前技术栈不变（Spring Boot + MySQL + Redis + Kafka + WebSocket）。
- 改造结果需可编译、可运行，并兼容现有前端消息交互方式。

## 2. 本次改造范围
- 后端消息链路（`message`、`im`、`relation` 相关模块）。
- 初始化 SQL（新增离线收件箱/投递状态表）。
- 备忘录中“消息正文单存储 + 投递状态轻量化 + ACK 状态推进”相关实现。

## 3. 任务分解
1. 数据模型升级
- 新增 `message_inbox` 表，按接收者记录消息投递状态（`SENT/DELIVERED/READ`）。
- 保持 `message` 表仅保存消息正文，避免群聊正文多副本。

2. 幂等与状态机
- 支持客户端传入 `msgId`，服务端优先复用该 `msgId` 做幂等写入。
- 消息 ACK 从“改消息主表状态”调整为“按接收者更新收件箱状态”。

3. 路由与离线补偿
- 私聊/群聊发送时，为接收方创建收件箱记录。
- 在线用户推送成功后标记 `DELIVERED`；离线用户保留 `SENT`，上线后通过离线接口补拉。

4. 离线消息查询
- `/api/v1/message/offline` 改为基于 `message_inbox` 拉取当前用户未读消息。
- 拉取后将 `SENT` 推进到 `DELIVERED`，由前端 ACK 再推进到 `READ`。

5. 验证与回归
- 执行 Maven 编译验证。
- 检查关键类诊断信息，确保未引入新的编译错误。

## 4. 验收标准
- 私聊/群聊发送后 `message` 仅新增一条正文记录。
- 离线用户存在 `message_inbox` 记录，状态初始为 `SENT`。
- 在线推送成功后状态可推进为 `DELIVERED`。
- 客户端 ACK 后，当前接收者状态推进为 `READ`，不影响其他接收者状态。
- 项目可正常编译通过。

## 5. 非目标说明
- 本次不实现跨机房双活、自动扩缩容、全文检索、复杂 Outbox 事务编排。
- 本次不重构前端架构，仅做必要兼容调整。





## 6. P0 剩余增强（按开发顺序）
1. Outbox 雏形（先落库、后发布）
- 新增 `message_outbox` 表：记录 `msg_id`、`target_user_id`、`topic`、发布状态、重试次数、下次重试时间。
- 发送链路在写 Kafka 前先创建 Outbox 事件；若即时发布成功则标记 `PUBLISHED`，失败则标记 `FAILED` 等待补偿任务。
- 增加定时发布器扫描待重试事件并补发，采用指数退避控制重试频率。

2. Kafka 重试主题消费补偿
- 新增重试消息载体（包含 `msgId`、`targetUserId`、`retryCount`、原始消息体）。
- 主消费主题投递失败时写入重试主题（`im-private-retry`、`im-group-retry`）。
- 重试消费者按最大重试次数补偿投递，超过阈值后记录失败并保留收件箱状态以供后续离线补拉。

3. 基础限流（用户维度）
- 在 WebSocket 发送入口增加按用户维度的滑动窗口/秒级计数限流（Redis `INCR + EXPIRE`）。
- 超限后直接拒绝发送并返回错误码/错误信息，不进入持久化和分发链路。
- 限流阈值与重试参数放入配置文件，支持环境调优。

## 7. P0 剩余增强验收标准
- Outbox 事件在 Kafka 瞬时异常下可被定时任务补发，且状态从 `FAILED` 变更为 `PUBLISHED`。
- Kafka 消费投递失败后可进入重试主题，重试成功后完成投递状态推进。
- 单用户超出阈值发送会被限流，系统整体稳定性不受单点高频发送影响。
