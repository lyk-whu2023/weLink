# WeLink 后端成员分工文档

## 1. 分工说明

本分工仅覆盖后端代码，按照当前项目的模块边界、文件职责和协作成本进行划分。

结合当前项目体量与耦合关系，建议按 3 名后端成员进行分工：

- 成员A：公共基础设施 + 用户认证
- 成员B：好友关系群组 + 消息存储会话
- 成员C：实时通信 + 可靠投递

分工原则如下：

- 成员A负责全局公共能力与登录认证，保证基础设施和鉴权链路统一。
- 成员B负责核心业务数据域，包括关系、群组、消息落库与会话查询。
- 成员C负责在线链路与异步可靠投递，集中处理高并发和跨实例问题。
- 测试文件跟随所属主模块分配，保证开发与验证责任一致。

## 2. 模块总览

当前后端主要由以下部分构成：

- 启动与配置：Spring Boot 启动类、Maven、配置文件、SQL、Redis/MinIO/JWT/MyBatis 等公共能力
- 用户认证：注册、登录、用户信息、JWT 登录态
- 好友与群组：好友申请、好友列表、群创建、加群、邀请、踢人、退群
- 消息存储与会话：消息落库、收件箱、历史消息、离线消息、未读会话、群消息序号
- 实时通信与可靠投递：WebSocket、在线状态、Kafka 消费、跨实例路由、Outbox 重试

## 4. 测试文件归属

为避免“功能由一人开发、测试由另一人维护”的情况，测试文件按业务归属划分如下：

- 用户认证相关：当前暂无独立测试文件，由成员A补充
- 关系群组与消息存储会话：`src/test/java/com/epsilon/welink/relation/RelationConcurrencyTest.java`、`src/test/java/com/epsilon/welink/message/ConcurrentGroupSeqTest.java`、`src/test/java/com/epsilon/welink/message/InboxStatusTransitionTest.java`
- 实时通信与可靠投递：`src/test/java/com/epsilon/welink/im/RateLimitConcurrencyTest.java`、`src/test/java/com/epsilon/welink/message/MessageConsistencyTest.java`
- 工程基础联通性：`src/test/java/com/epsilon/welink/WeLinkApplicationTests.java` 由成员A负责维护

## 5. 建议协作规则

- 涉及 `common`、`gateway`、`application.properties`、`pom.xml` 的改动，默认先同步成员A。
- 涉及 `MessageService.java` 与 `IMService.java` 联动的改动，成员B和成员C必须共同评审。
- 涉及群成员读游标 `lastReadSeq`、群消息序号 `groupSeq` 的改动，成员B和成员C必须共同确认。
- 涉及登录态、JWT、请求鉴权的改动，成员A和成员C必须共同确认。
- 新增测试时，优先由对应模块负责人补充，不跨模块随意修改对方测试。

## 6. 最终分工结论

按照当前代码结构，最合理的后端分工方式是：

- 成员A负责公共基础设施、项目级配置、文件存储和用户认证
- 成员B负责好友关系、群组、消息存储、会话、收件箱和历史消息
- 成员C负责 WebSocket、在线状态、Kafka、Outbox 和可靠投递

该划分能够保证：

- 控制层、服务层、实体层、Mapper 层基本按业务域归属统一
- 跨模块高频协作点集中在少数接口边界，冲突较少
- 并发一致性、消息可靠性、在线链路这些高风险部分有明确负责人
