# WeLink 后端开发先后顺序文档

## 1. 说明

本文档基于当前的 3 人后端分工，给出一个更适合落地执行的开发先后顺序。

排序原则如下：

- 先做所有模块依赖的基础设施和运行环境
- 再做业务基础数据域，保证用户、关系、群组、消息表结构和接口可用
- 最后接入实时通信、Kafka、Outbox 和并发一致性优化
- 每个阶段尽量保证“有明确负责人、有对应文件、有可交付结果”

当前成员分工如下：

- 成员A：公共基础设施 + 用户认证
- 成员B：好友关系群组 + 消息存储会话
- 成员C：实时通信 + 可靠投递

## 2. 总体顺序

建议按以下顺序推进：

1. 第1阶段：项目底座与运行环境
2. 第2阶段：用户与认证
3. 第3阶段：好友关系与群组
4. 第4阶段：消息存储与会话查询
5. 第5阶段：实时通信与可靠投递
6. 第6阶段：联调、压测与测试补齐

## 4. 并行开发建议

为了提高效率，建议按下面方式并行推进：

- 第1阶段由成员A先完成，成员B、成员C在阶段末参与环境验证
- 第2阶段开始后，成员B可以先建关系表、群组表、Mapper 和实体
- 第4阶段前半段可由成员B先完成消息表、收件箱表和基础 Service
- 第5阶段前半段成员C可以先搭好 WebSocketServer、ChannelHandler 和 Kafka 消费框架
- 成员C正式接入实时投递前，必须等成员A的 JWT 能力和成员B的消息落库能力稳定

## 5. 推荐执行结论

如果按最稳妥的方式推进，推荐顺序是：

1. 成员A先完成项目底座和用户认证
2. 成员B随后完成好友关系、群组和消息存储会话
3. 成员C最后完成实时通信、Kafka、Outbox 和可靠投递
4. 三人一起完成联调、测试和并发验证

这个顺序的优势是：

- 依赖关系最清晰
- 中途返工最少
- 实时链路不会因为底层数据结构未稳定而反复修改
