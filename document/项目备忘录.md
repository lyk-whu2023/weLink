# WeLink 分布式消息系统技术备忘录

## 1. 文档目标与范围
- 本备忘录用于明确 WeLink 在分布式场景下的消息系统设计方案，重点覆盖因果一致性、幂等性、存储与查询、消息处理链路以及高并发高可用优化策略。
- 本文档聚焦后端消息链路与中间件协同设计，不涉及前端交互细节与原型实现。
- 目标是在保证消息正确性的前提下，实现低延迟、可扩展、可观测、可运维的工程化落地方案。

## 2. 总体架构设计
### 2.1 设计原则
- 正确性优先：先保证不丢、不重、可追踪，再优化吞吐与时延。
- 解耦与异步：连接层、存储层、分发层职责分离，通过 Kafka 解耦峰值流量。
- 最终一致：采用“落库 + 异步投递 + ACK 状态推进 + 补偿重试”保证最终一致。
- 面向扩展：支持多实例横向扩容、热点群隔离、数据冷热分层。

### 2.2 组件职责
- MySQL：消息主存储、历史查询、离线收件箱、投递状态与审计数据。
- Redis：在线状态、路由表、短期幂等去重、热点会话缓存。
- Kafka：跨实例消息分发、削峰填谷、异步重试通道。
- WebSocket 网关：连接管理、实时推送、ACK 回传、心跳与重连控制。

### 2.3 私聊与群聊处理模型
- 私聊：消息先持久化，再按接收方在线状态路由；同实例直推，跨实例 Kafka 转发，离线写入收件箱。
- 群聊：采用“逻辑写扩散 + 在线分流 + 离线补偿”模型；消息正文仅存一份，离线成员以轻量收件箱记录补发。

## 3. 设计一：因果一致性与顺序保证
### 3.1 目标
- 保证群聊消息在同一会话中的可解释顺序，避免跨实例并发导致乱序展示。

### 3.2 实现方案
1. 群内单调序号：每个 `groupId` 维护全局递增 `groupSeq`（Redis `INCR`），并周期性落盘到 MySQL。
2. 分区有序消费：Kafka 以 `groupId` 作为分区键，同群消息固定路由同一分区。
3. 展示序与到达序分离：客户端按 `groupSeq` 排序展示，不以网络到达时间作为展示依据。
4. 缺口补偿机制：若客户端检测到序号缺口，调用“按序号区间补拉”接口进行补齐。
5. 多端读状态统一：用户维度维护“已读最大 `groupSeq`”，多端共享读游标。

### 3.3 风险与控制
- 风险：`groupSeq` 分配失败导致发送阻塞。
- 控制：短暂失败快速重试，超时进入降级队列并触发告警，不直接丢弃消息。

## 4. 设计二：消息幂等性与可靠投递
### 4.1 目标
- 保证消息在重试、重复提交、重复消费场景下“最多一次生效结果”，同时维持链路“至少一次投递”能力。

### 4.2 实现方案
1. 全局消息标识：客户端或网关生成 `msgId`，`message.msg_id` 建唯一索引。
2. 发送幂等：客户端重试必须复用同一 `msgId`；写库冲突视为已成功受理。
3. 投递幂等：使用 `deliver:{receiverId}:{msgId}` 作为去重键（Redis + MySQL 状态表双层兜底）。
4. ACK 状态机：`SENT -> DELIVERED -> READ` 单向推进，重复 ACK 不回退。
5. 消费幂等：Kafka 消费前检查幂等记录，重复消息直接跳过。
6. 重试补偿：失败消息进入指数退避队列，超阈值由定时任务补偿并记录异常事件。

## 5. 设计三：存储模型与查询能力
### 5.1 目标
- 在高并发场景下控制存储成本与查询延迟，避免“群聊正文多副本”带来的写放大。

### 5.2 数据模型建议
1. `message`（主表）：每条消息仅存一份正文，字段含 `msg_id`、会话维度、`group_seq`、内容与时间戳。
2. `message_inbox`（收件箱）：按接收者记录投递状态，不复制正文，支持离线补发与状态追踪。
3. 可选 `message_delivery`（投递审计）：用于精准记录多次投递尝试、失败原因与重试轨迹。

### 5.3 索引与分页策略
1. 私聊索引：`(conversation_id, created_at)` 或 `(to_user_id, created_at)`。
2. 群聊索引：`(group_id, group_seq)`，用于严格顺序分页。
3. 查询统一游标分页，避免深度 `offset` 造成慢查询。

### 5.4 数据分层与生命周期
- 热数据保留在主表高性能索引区间（如近 30 天）。
- 冷数据按月分表或归档，查询通过冷热路由完成透明访问。
- Redis 缓存仅用于加速，不作为消息真相来源。

## 6. 设计四：消息链路实现（在线/离线，私聊/群聊）
### 6.1 私聊发送链路
1. 客户端发送消息并携带 `msgId`。
2. 网关鉴权与参数校验通过后，消息先落库。
3. 查询目标用户在线状态与路由实例。
4. 同实例在线：WebSocket 直推。
5. 跨实例在线：Kafka 转发至目标实例后推送。
6. 目标离线：写入 `message_inbox`，待上线补发。

### 6.2 群聊发送链路
1. 校验发送权限，申请 `groupSeq`，写入 `message` 主记录。
2. 获取成员列表并剔除发送者。
3. 在线成员实时推送（同实例直推、跨实例转发）。
4. 离线成员批量写入 `message_inbox`。

### 6.3 上线补偿与失败处理
- 用户重连后优先拉取未确认消息，再进行实时流合并。
- 推送失败先短重试，后转离线补偿；定时任务扫描超时未 ACK 记录触发补投递。
- 全链路日志绑定 `traceId/msgId/groupSeq`，支持端到端问题定位。

## 7. 补充建议（后续开发设计，按优先级）
### P0（上线必备）
- 消息可靠性：落库优先 + Outbox + 重试补偿，消除“落库成功但未投递”窗口。
- 幂等闭环：`msgId` 唯一、投递去重、消费幂等、ACK 单向状态机全部落地。
- 接入高可用：WebSocket 网关多实例，故障摘除与连接迁移机制可用。
- 存储治理：正文单存储、离线状态轻量化、关键索引完备。
- 流量保护：用户/群限流与基础降级策略可执行。
- 基础可观测：发送成功率、P95 时延、Kafka 积压、重试率四类核心指标可监控。

### P1（性能增强）
- 链路削峰：Kafka 批量、压缩、并行消费；私聊与群聊 Topic 隔离。
- 大群优化：超过阈值切换“在线直推 + 离线拉取”，并实施分片并行推送。
- 查询优化：统一游标分页，热点会话缓存最近消息。
- 线程模型：Netty I/O 与业务线程池隔离，心跳参数按规模调优。
- 同城容灾：MySQL 主从切换、Redis 哨兵/集群、Kafka 多副本达标。

### P2（规模化演进）
- 冷热分层：消息分区分表与归档体系完善。
- 跨 AZ/机房容灾：关键链路双活或主备，形成可演练切换机制。
- 弹性扩缩容：基于连接数、时延、积压量的自动扩缩容策略。
- 高级 SLO 治理：P99、错误预算、分级告警、值班闭环。
- 检索扩展：异步构建全文检索链路（如 ES），与主写链路解耦。

## 8. 验收指标建议（用于阶段评审）
- 可靠性：消息最终送达率 >= 99.99%，重复生效率 <= 0.01%。
- 时延：私聊端到端 P95 < 500ms，群聊端到端 P95 < 800ms。
- 可用性：消息服务月度可用性 >= 99.9%。
- 补偿效率：离线补发 P95 完成时延 < 3s（上线后）。
