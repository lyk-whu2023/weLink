# WeLink 项目设计文档

## 1. 项目愿景

构建一个具备**高并发、高可用**能力的分布式即时通讯系统（简化版 QQ），支持海量用户在线互动、即时聊天。项目初期聚焦于**私聊和群聊核心功能**的实现，采用**单体架构**但考虑分布式部署场景，前端采用 **Web 页面**形式，后续通过浏览器嵌套方式实现客户端。

***

## 2. 系统架构设计

### 2.1 整体架构

采用**单体架构 (Monolithic Architecture)**，将核心功能模块整合在单一应用中，但通过合理的模块划分和接口设计，为后续可能的分布式演进预留空间。

```
┌─────────────────────────────────────────────────────────┐
│                    客户端 (Web Browser)                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │  Vue/React│  │ WebSocket│  │  REST API │              │
│  └──────────┘  └──────────┘  └──────────┘              │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                   Nginx 负载均衡层                        │
│         (反向代理 + 负载均衡 + SSL 终结)                  │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                  WeLink 后端单体应用                       │
│  ┌──────────────────────────────────────────────────┐  │
│  │              API Gateway 模块                      │  │
│  │        (路由分发、鉴权、限流、跨域处理)              │  │
│  └──────────────────────────────────────────────────┘  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌───────┐  │
│  │ 用户模块  │  │ 关系模块  │  │ 消息模块  │  │IM模块 │  │
│  │  User    │  │Relation  │  │ Message  │  │ IM    │  │
│  └──────────┘  └──────────┘  └──────────┘  └───────┘  │
│  ┌──────────────────────────────────────────────────┐  │
│  │              公共服务层                            │  │
│  │        (文件存储、缓存服务、消息队列消费者)          │  │
│  └──────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
                          │
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼

   │ MySQL   │      │ Redis   │      │ Kafka   │
   │ 8.0     │      │ Cluster │      │         │
   └─────────┘      └─────────┘      └─────────┘
```

### 2.2 模块划分

| 模块名称                | 包路径                   | 核心职责                        | 数据归属      |
| :------------------ | :-------------------- | :-------------------------- | :-------- |
| **Gateway Module**  | `com.welink.gateway`  | 统一入口、鉴权(JWT)、限流、跨域处理        | 无         |
| **User Module**     | `com.welink.user`     | 用户注册、登录、个人信息管理              | 用户表       |

| **IM Module**       | `com.welink.im`       | WebSocket 长连接管理、心跳检测、消息实时推送 | Redis 路由表 |
| **Common Module**   | `com.welink.common`   | 文件存储、工具类、通用配置、异常处理          | 无         |

### 2.3 分布式部署考虑

虽然是单体架构，但需要考虑以下分布式部署场景：

#### 2.3.1 多实例部署

```
                    Nginx (负载均衡)
                   /      |      \
              实例1     实例2     实例3
              (IM)     (IM)     (IM)
```

**关键问题与解决方案：**

| 问题                 | 解决方案                                               |
| :----------------- | :------------------------------------------------- |
| **WebSocket 会话共享** | Redis 存储 `UserId -> ServerInstanceId` 映射，实现跨实例消息路由 |
| **Session 一致性**    | JWT 无状态认证 + Redis 存储用户会话状态                         |
| **分布式锁**           | Redisson 实现分布式锁，防止并发操作冲突                           |
| **消息顺序性**          | 同一会话的消息通过 Kafka Partition Key 保证顺序                 |
| **配置一致性**          | Nacos 配置中心统一管理配置，支持动态刷新                            |

#### 2.3.2 会话路由设计

```
用户 A 发送消息给用户 B
         │
         ▼
    当前实例检查
         │
    ┌────┴────┐
    │         │
  B在线      B离线
  同实例      │
    │        ▼
    │     存储离线消息
    ▼        │
  直接推送   │
            │
  B在线不同实例
    │
    ▼
  查询 Redis 路由表
    │
    ▼
  通过 Kafka 转发到目标实例
    │
    ▼
  目标实例推送给 B
```

## 4. 数据库设计

### 4.1 核心表结构

#### 用户表 (user)

```sql
CREATE TABLE `user` (
  `id` BIGINT PRIMARY KEY AUTO_INCREMENT,
  `username` VARCHAR(50) UNIQUE NOT NULL,
  `password` VARCHAR(100) NOT NULL,
  `nickname` VARCHAR(50),
  `avatar` VARCHAR(255),
  `email` VARCHAR(100),
  `phone` VARCHAR(20),
  `status` TINYINT DEFAULT 1 COMMENT '1-正常 0-禁用',
  `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP,
  `updated_at` DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  INDEX `idx_username` (`username`),
  INDEX `idx_email` (`email`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 好友关系表 (friend\_relation)

```sql
CREATE TABLE `friend_relation` (
  `id` BIGINT PRIMARY KEY AUTO_INCREMENT,
  `user_id` BIGINT NOT NULL,
  `friend_id` BIGINT NOT NULL,
  `status` TINYINT DEFAULT 0 COMMENT '0-待确认 1-已好友 2-已拒绝',
  `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP,
  `updated_at` DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY `uk_user_friend` (`user_id`, `friend_id`),
  INDEX `idx_user_id` (`user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 群组表 (group\_info)

```sql
CREATE TABLE `group_info` (
  `id` BIGINT PRIMARY KEY AUTO_INCREMENT,
  `group_name` VARCHAR(100) NOT NULL,
  `avatar` VARCHAR(255),
  `owner_id` BIGINT NOT NULL,
  `notice` TEXT COMMENT '群公告',
  `member_count` INT DEFAULT 0,
  `status` TINYINT DEFAULT 1 COMMENT '1-正常 0-解散',
  `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP,
  `updated_at` DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  INDEX `idx_owner_id` (`owner_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 群成员表 (group\_member)

```sql
CREATE TABLE `group_member` (
  `id` BIGINT PRIMARY KEY AUTO_INCREMENT,
  `group_id` BIGINT NOT NULL,
  `user_id` BIGINT NOT NULL,
  `role` TINYINT DEFAULT 0 COMMENT '0-普通成员 1-管理员 2-群主',
  `join_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY `uk_group_user` (`group_id`, `user_id`),
  INDEX `idx_group_id` (`group_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 消息表 (message)

```sql
CREATE TABLE `message` (
  `id` BIGINT PRIMARY KEY AUTO_INCREMENT,
  `msg_id` VARCHAR(64) UNIQUE NOT NULL COMMENT '消息唯一ID',
  `from_user_id` BIGINT NOT NULL,
  `to_user_id` BIGINT COMMENT '目标用户ID(私聊)',
  `group_id` BIGINT COMMENT '群组ID(群聊)',
  `msg_type` TINYINT NOT NULL COMMENT '1-文本 2-图片 3-文件 4-系统',
  `content` TEXT NOT NULL,
  `status` TINYINT DEFAULT 0 COMMENT '0-未读 1-已读',
  `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP,
  INDEX `idx_from_user` (`from_user_id`),
  INDEX `idx_to_user` (`to_user_id`),
  INDEX `idx_group_id` (`group_id`),
  INDEX `idx_created_at` (`created_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 4.2 分库分表策略

当数据量增长时，考虑以下分片策略：

| 表                | 分片键                       | 策略         |
| :--------------- | :------------------------ | :--------- |
| message          | `to_user_id` 或 `group_id` | 按用户/群组哈希分表 |
| friend\_relation | `user_id`                 | 按用户哈希分表    |
| group\_member    | `group_id`                | 按群组哈希分表    |

## 5. 核心功能设计

### 5.1 用户系统

#### 5.1.1 注册/登录流程

```
注册流程:
用户输入信息 → 参数校验 → 唯一性检查 → 密码加密(BCrypt) → 入库 → 返回成功

登录流程:
用户输入账号密码 → 查询用户 → 密码校验 → 生成JWT → 缓存用户信息到Redis → 返回Token
```

#### 5.1.2 JWT 认证设计

```json
{
  "header": {
    "alg": "HS256",
    "typ": "JWT"
  },
  "payload": {
    "userId": 123456,
    "username": "user123",
    "exp": 1700000000,
    "iat": 1699996400
  }
}
```

**Token 管理策略：**

- Access Token: 有效期 2 小时
- Refresh Token: 有效期 7 天，存储于 Redis
- Token 黑名单：用户登出时将 Token 加入 Redis 黑名单

### 5.2 好友系统

#### 5.2.1 好友申请流程

```
A 发送好友申请 → 检查是否已是好友 → 创建申请记录 → 推送通知给 B
                                                    │
                                        ┌───────────┴───────────┐
                                        │                       │
                                    B 接受                   B 拒绝
                                        │                       │
                                        ▼                       ▼
                                  更新状态为已好友          更新状态为已拒绝
                                  双向添加好友记录          通知 A 被拒绝
```

#### 5.2.2 好友列表查询

```sql
-- 查询用户的好友列表
SELECT u.id, u.username, u.nickname, u.avatar, u.status
FROM friend_relation fr
JOIN user u ON fr.friend_id = u.id
WHERE fr.user_id = ? AND fr.status = 1
ORDER BY fr.updated_at DESC;
```

### 5.3 群组系统

#### 5.3.1 创建群组流程

```
用户发起创建 → 填写群信息 → 创建群记录 → 添加创建者为群主 → 邀请成员 → 批量添加群成员
```

#### 5.3.2 群成员管理

| 操作    | 权限     | 说明        |
| :---- | :----- | :-------- |
| 邀请成员  | 群主/管理员 | 发送邀请通知    |
| 踢出成员  | 群主/管理员 | 移除群成员记录   |
| 退出群组  | 所有成员   | 群主退出需转移权限 |
| 修改群信息 | 群主/管理员 | 群名、公告等    |

### 5.4 即时通讯核心

#### 5.4.1 WebSocket 连接管理

```
客户端连接 → WebSocket 握手 → 发送认证 Token → 服务端验证 → 建立连接
                                                        │
                                              ┌─────────┴─────────┐
                                              │                   │
                                        记录路由表到 Redis      启动心跳检测
                                        UserId -> InstanceId    60s 超时断开
```

#### 5.4.2 消息发送流程

**私聊消息：**

```
A 发送消息 → 生成 msgId → 持久化到 MySQL → 推送 Kafka → 查询 B 在线状态
                                                              │
                                                  ┌───────────┴───────────┐
                                                  │                       │
                                              B 在线                  B 离线
                                                  │                       │
                                        ┌─────────┴─────────┐            │
                                        │                   │            ▼
                                  同实例              不同实例        标记为离线消息
                                  直接推送            Kafka 转发
```

**群聊消息：**

```
A 发送群消息 → 生成 msgId → 持久化到 MySQL → 推送 Kafka → 查询群成员列表
                                                              │
                                                              ▼
                                                    过滤在线成员列表
                                                              │
                                                              ▼
                                              ┌───────────────┼───────────────┐
                                              │               │               │
                                          同实例成员      不同实例成员      离线成员
                                          直接推送        Kafka 转发       标记离线
```

#### 5.4.3 消息可靠性保障

| 机制        | 实现方式                          |
| :-------- | :---------------------------- |
| **消息不丢失** | MySQL 持久化 + Kafka 备份 + ACK 确认 |
| **消息不重复** | msgId 唯一索引 + 客户端去重            |
| **消息顺序性** | 同一会话的消息按时间戳排序                 |
| **已读回执**  | 客户端发送已读确认 → 更新消息状态            |
| **离线消息**  | 用户上线时拉取未读消息列表                 |

#### 5.4.4 心跳检测机制

```
客户端 → 每 30s 发送 Ping → 服务端回复 Pong
                                │
                          记录最后心跳时间
                                │
                          60s 未收到 Ping → 断开连接
                                │
                          清理路由表 → 更新在线状态
```

### 5.5 在线状态管理

```
用户上线 → WebSocket 连接成功 → Redis 设置在线状态
                                    │
                          SET user:online:{userId} {instanceId} EX 120
                                    │
                          发布订阅通知好友上线状态
```

**状态同步策略：**

- 使用 Redis Hash 存储在线状态：`user:online:{userId}` → `instanceId`
- 设置过期时间 120 秒，心跳续期
- 通过 Redis Pub/Sub 通知好友状态变化

## 6. 分布式问题解决方案

### 6.1 会话路由问题

**问题：** 多实例部署时，如何找到目标用户所在的实例？

**解决方案：**

```java
// Redis 路由表结构
// Key: im:route:{userId}
// Value: {instanceId}
// TTL: 120s

// 发送消息时查询路由
String targetInstance = redisTemplate.opsForValue().get("im:route:" + userId);
if (targetInstance != null && !targetInstance.equals(currentInstanceId)) {
    // 跨实例消息，通过 Kafka 转发
    kafkaTemplate.send("im-message-route", userId, message);
}
```

### 6.2 分布式锁问题

**问题：** 多实例并发操作同一资源时如何保证一致性？

**解决方案：** 使用 Redisson 实现分布式锁

```java
// 好友申请加锁
RLock lock = redissonClient.getLock("friend:apply:" + userId);
try {
    lock.lock(10, TimeUnit.SECONDS);
    // 执行好友申请逻辑
} finally {
    lock.unlock();
}

// 群成员操作加锁
RLock lock = redissonClient.getLock("group:member:" + groupId);
```

### 6.3 消息顺序性问题

**问题：** 多实例处理消息时如何保证同一会话的消息顺序？

**解决方案：**

- Kafka Partition Key 使用 `conversationId`（会话ID）
- 同一会话的消息路由到同一 Partition
- 消费者单线程消费同一 Partition

```java
// 发送消息到 Kafka
String conversationKey = getConversationKey(fromUserId, toUserId, groupId);
kafkaTemplate.send("im-message", conversationKey, message);
```

### 6.4 缓存一致性问题

**问题：** 数据库更新后如何保证缓存一致性？

**解决方案：** Cache Aside Pattern

```
更新数据 → 先更新数据库 → 再删除缓存
读取数据 → 先读缓存 → 缓存未命中则读数据库 → 写入缓存
```

### 6.5 分布式事务问题

虽然不使用 Seata，但通过以下方式保证最终一致性：

| 场景   | 解决方案              |
| :--- | :---------------- |
| 发送消息 | 先落库再推 Kafka，失败则重试 |
| 创建群组 | 本地事务保证群信息和成员数据一致性 |
| 好友申请 | 本地事务保证申请记录和通知一致性  |

```json
    "content": "Hello",
    "timestamp": 1700000000000
  }
}
```

```typescript
class IMClient {
  private ws: WebSocket;
  private reconnectTimer: number;
  
  connect(token: string) {
    this.ws = new WebSocket(`ws://host/ws/im?token=${token}`);
    this.ws.onmessage = this.handleMessage;
    this.ws.onclose = this.handleReconnect;
    this.startHeartbeat();
  }
  
  sendMessage(message: Message) {
    this.ws.send(JSON.stringify({
      type: 'message',
      data: message
    }));
  }
  
  startHeartbeat() {
    setInterval(() => {
      this.ws.send(JSON.stringify({ type: 'heartbeat' }));
    }, 30000);
  }
}
```

## 11. 难点与解决方案

| 难点                | 解决方案                                               |
| :---------------- | :------------------------------------------------- |
| **IM 分布式会话路由**    | Redis 存储 `UserId -> InstanceId` 映射，Kafka 实现跨实例消息转发 |
| **消息不丢失**         | MySQL 持久化 + Kafka 备份 + ACK 确认机制 + 重试机制             |
| **消息顺序性**         | Kafka Partition Key 按会话路由，保证同会话消息顺序消费              |
| **在线状态同步**        | Redis Hash 存储 + TTL 过期 + 心跳续期 + Pub/Sub 通知         |
| **WebSocket 连接数** | Netty 优化 + 连接池 + 负载均衡 + 水平扩展                       |
| **群消息扩散性能**       | 批量查询在线成员 + 异步推送 + 离线消息延迟处理                         |
| **热点群问题**         | 大群消息采用写扩散模式，提前计算推送列表                               |
| **前端重连机制**        | WebSocket 断线自动重连 + 消息补偿拉取                          |
